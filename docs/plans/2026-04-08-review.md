## NON-BLOCKING (May Be Deferred)

**Hardcoded enum names in macros (Keyword, Punctuation, Literal):**
- Description: The `keywords!` macro always generates an enum named `Keyword`, `punctuation!` always `Punctuation`, `literals!` always `Literal`. If a user calls two `keywords!` invocations in the same scope, they get a name collision. Consider allowing an optional enum name parameter.
- Location: recursa-core/src/macros.rs:31, :64, :94
- Action: Could add an optional first parameter for the enum name, e.g. `keywords!(MyKeywords { Let => "let" })`. Low priority since the current API matches the plan and single-invocation-per-scope is the expected pattern.

**`bulk_macros.rs` test uses `recursa_core` directly instead of `recursa`:**
- Description: The test file imports `recursa_core` directly and calls `recursa_core::keywords!`. While this works and validates the macros at the core crate level, it differs from how downstream users will consume the library (via `recursa`). The `integration.rs` test already validates the `recursa::keywords!` path.
- Location: tests/bulk_macros.rs:1-3
- Action: Consider changing to `use recursa::{Input, NoRules, Scan}` and `recursa::keywords!` for consistency, or keep both paths tested intentionally (current approach tests both paths, which has value).

**`regex` re-exported implicitly but not explicitly:**
- Description: The facade re-exports `miette` explicitly (`pub use miette;`) but `regex` is only a dependency, not re-exported. Users who want to use `regex` directly still need their own dependency. Since the derive macro generates `::regex::Regex` references, having `regex` as a dependency is sufficient for compilation, but explicit re-export would be more consistent with the `miette` treatment.
- Location: src/lib.rs
- Action: Consider adding `pub use regex;` for symmetry with `pub use miette;`, or document that `regex` is intentionally not re-exported since users rarely need it directly.
